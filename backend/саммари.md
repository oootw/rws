# Backend — саммари для внедрения фич

Документ-шпаргалка для AI-агента, который дописывает фичи в Guard Reviews.
Читать вместе с `backend/docs/architecture.md` (там — принципы и правила слоёв).
Здесь — конкретные рецепты, имена, пути.

---

## 1. Стек

- **PHP 8.4**, **Laravel 13**, **Pest 4** для тестов.
- **Sanctum** (auth, пока минимально используется), **Nutgram** (Telegram-бот, auto-discovered).
- **PostgreSQL** через Eloquent (UUID PK у всех агрегатов).
- **Redis** очереди (`SendNegativeReviewAlert`, `SendWeeklyDigest`, `SendSubscriptionReminder`).
- **Tinkoff** — единственный эквайер.
- Капча — Yandex SmartCaptcha (контракт `App\Contracts\Captcha\CaptchaVerifierInterface`, в тестах подменяется на `NullCaptchaVerifier`).
- Запуск локально: `make local-up` или `composer dev` внутри `backend/`. Тесты: `composer test`.

## 2. Архитектурная карта

Зависимости направлены строго внутрь: **Interface → Application → Domain**.
`Infrastructure` реализует интерфейсы из `Domain` и `Application`.

```
app/
├── Domain/            ← pure PHP, без Laravel
│   └── <Context>/     (агрегаты, VO, enums, события, *Repository интерфейсы)
├── Application/       ← use cases (Command/Query + Handler)
│   └── <Context>/<UseCase>/  (один use case = одна папка)
├── Infrastructure/    ← реализации: Eloquent, Tinkoff, Telegram, каналы доставки
│   ├── Persistence/Eloquent/<Context>/  (репозитории, мапперы, ридеры)
│   └── ServiceProviders/<Context>ServiceProvider.php
├── Interface/         ← entry points
│   ├── Http/Controllers/{Public,Webhook,Internal}/
│   ├── Http/Requests/Public/
│   ├── TelegramBot/{Commands,Conversations,Callbacks,Middleware,Support}/
│   └── Console/Commands/
├── Http/              ← Laravel-стандартные middleware и legacy controllers
│   ├── Controllers/Api/Public/   (старые публичные API, тонкие)
│   └── Middleware/   (tenant, subscription.active, resolve.public.place)
├── Models/            ← Eloquent-модели (только в Infrastructure-слое по смыслу)
├── Jobs/              ← async-адаптеры очередей (склейка, без бизнес-логики)
├── Enums/, Support/, Mail/, Services/, Contracts/
```

| Context         | Агрегат           | Ключевые use cases                                                              |
|-----------------|-------------------|---------------------------------------------------------------------------------|
| `Iam`           | `Owner`           | RegisterOwner, ResolveTenantBySubdomain, GetOwnerById, GetOwnerByTelegram, ExtendSubscription, CalculateSubscriptionAmount, UpdateOwnerProfile, DeleteOwner, OverrideSubscription, ChangeOwnerTariff, IssueOwnerImpersonationToken, **SetDefaultTariff** |
| `Places`        | `Place`           | RegisterPlace, GetPlaceForOwner, ListOwnerPlaces, ResolvePublicPlace, GetPublicPlaceView, **UpdatePlace**, **ChangePlaceActivation**, **DeletePlace** |

| `Payments`      | `PaymentTransaction` | InitSubscriptionPayment, HandlePaymentNotification, **ForceFailPayment**     |
| `Notifications` | —                 | NotifyAboutNegativeReview, ConfirmSubscriptionRenewed, RemindAboutSubscriptionExpiry, AlertAboutCriticalError, SendWeeklyDigest |
| `TelegramBot`   | —                 | Команды, conversations, callbacks Nutgram                                       |

Cross-context общение **только через use cases** или Domain Events. Eloquent-модели чужого контекста дёргать **нельзя**.

**Admin-панель** (Filament v5, `/admin`) — отдельный Interface-слой `app/Filament/Resources/<Context>/`. Resources привязаны к persistence-моделям, но **все мутации идут через Application use cases**. Подробнее — `backend/админка-план.md`. Исключение: конфигурационные сущности без сложных инвариантов (Tariff) — CRUD напрямую через Eloquent.

**Owner-панель** (React SPA, `{slug}.otziv.space/owner`) — отдельный фронт в `frontend/owner/` (Vite + React + Tailwind + React Query + Zustand + PWA, стиль Kvell Merchant). Бэк отдаёт SPA-shell через `OwnerSpaController` под middleware `tenant`; API под `/api/owner/*` будет защищён Sanctum guard `owner` (cookie-сессия). Подробнее — `backend/docs/owner-panel-plan.md`. **Не путать с админкой Filament** — это разные интерфейсы для разных аудиторий: Filament для dev/админа, Owner SPA для конечных владельцев заведений.

## 3. Сквозные принципы — что НЕЛЬЗЯ нарушать

- **Domain слой не знает про Laravel.** Никаких `use Illuminate\…`, `now()`, `config()`, фасадов, `Model::*`. Время — через `Clock` (порт). ID — через `*IdGenerator` (порт).
- **`declare(strict_types=1);`** в каждом файле слоёв Domain/Application/Infrastructure.
- **`final readonly`** для VO, Command, Query, Handler. Агрегат — `final` (но не readonly, у него есть мутируемые поля).
- **Один Use Case = одна папка.** Внутри: `<UseCase>Command.php` (или `Query`), `<UseCase>Handler.php`, опционально `<UseCase>Result.php`.
- **Handler — `final readonly class` с публичным методом `handle()`.** Никаких других публичных методов.
- **VO валидируется в конструкторе** (`InvalidArgumentException`). Инварианты агрегата — в фабриках (`DomainException`).
- **Контроллер строго тонкий**: достать из request → собрать Command → вызвать handler → вернуть ответ. Никаких `DB::`, никакой бизнес-логики.
- **Eloquent-модель = деталь персистентности.** Никто кроме `EloquentXxxRepository` и `XxxMapper` (в `Infrastructure/Persistence/Eloquent/<Context>/`) её не видит.
- **Транзакции** — только через порт `App\Application\Shared\Transactions\TransactionRunner` (реализация: `EloquentTransactionRunner`).
- **События** — только через порт `App\Application\Shared\Events\DomainEventDispatcher` (реализация: `LaravelDomainEventDispatcher`). Listener регистрируется в `boot()` соответствующего ServiceProvider.

## 4. Шаблоны (копировать и адаптировать)

### 4.1. Доменный агрегат

```php
// app/Domain/<Context>/<Aggregate>.php
final class Foo
{
    /** @var list<DomainEvent> */
    private array $recordedEvents = [];

    private function __construct(
        public readonly FooId $id,
        // ... readonly публичные VO
        private SomeState $state,   // мутируемое — приватно, доступ через методы
    ) {}

    public static function create(FooId $id, /* VOs */): self { /* инвариант + record() */ }
    public static function restore(/* всё */): self { /* без событий */ }

    public function pullRecordedEvents(): array { /* как в Review.php */ }
    private function record(DomainEvent $event): void { $this->recordedEvents[] = $event; }
}
```

См. `app/Domain/Reviews/Review.php` — каноничный пример.

### 4.2. Value Object

```php
// app/Domain/<Context>/<Name>.php
final readonly class Stars
{
    private const MIN = 1; private const MAX = 5;
    public function __construct(public int $value)
    {
        if ($value < self::MIN || $value > self::MAX) {
            throw new InvalidArgumentException("…");
        }
    }
}
```

### 4.3. Use Case

```php
// app/Application/<Context>/<UseCase>/<UseCase>Command.php
final readonly class SubmitFooCommand
{
    public function __construct(
        public string $somethingId,
        public int $value,
        public ?string $optional,
    ) {}
}

// app/Application/<Context>/<UseCase>/<UseCase>Handler.php
final readonly class SubmitFooHandler
{
    public function __construct(
        private FooRepository $foos,
        private FooIdGenerator $idGenerator,
        private Clock $clock,
        private DomainEventDispatcher $events,
    ) {}

    public function handle(SubmitFooCommand $command): void
    {
        $foo = Foo::create(/* VO from command */);
        $this->foos->save($foo);
        $this->events->dispatchAll($foo->pullRecordedEvents());
    }
}
```

Если возвращается DTO — это `Query`+`QueryHandler`+`*View`/`*Result` (см. `ListRecentReviewsForOwner`).

### 4.4. Репозиторий

```php
// app/Domain/<Context>/<Aggregate>Repository.php  (интерфейс)
interface FooRepository {
    public function save(Foo $foo): void;
    public function findById(FooId $id): ?Foo;
}

// app/Infrastructure/Persistence/Eloquent/<Context>/Eloquent<Aggregate>Repository.php
final readonly class EloquentFooRepository implements FooRepository
{
    public function __construct(private FooMapper $mapper) {}
    public function save(Foo $foo): void {
        $model = FooModel::query()->find($foo->id->value) ?? new FooModel;
        $this->mapper->toPersistence($foo, $model)->save();
    }
    public function findById(FooId $id): ?Foo {
        $m = FooModel::query()->find($id->value);
        return $m === null ? null : $this->mapper->toDomain($m);
    }
}
```

**Биндинг** в `app/Infrastructure/ServiceProviders/<Context>ServiceProvider.php` в массиве `$bindings` (он распознаётся Laravel автоматически).

### 4.5. Read-Model (CQRS-light)

Для списков и чтений, где грузить агрегат — оверкилл, делаем отдельный `Reader`-порт в `Application/<Context>/<Query>/` и реализацию в `Infrastructure/Persistence/Eloquent/<Context>/Eloquent<X>Reader.php`. См. `RecentReviewsReader` / `EloquentRecentReviewsReader`.

### 4.6. Доменное событие

```php
// app/Domain/<Context>/Events/<Verb><Aggregate>.php
final readonly class FooHappened implements DomainEvent
{
    public function __construct(public FooId $fooId, public DateTimeImmutable $at) {}
}
```

В агрегате: `$this->record(new FooHappened(...));`
Слушатель — в `app/Application/<Context>/Listeners/<Name>.php`, регистрация в `boot()` сервис-провайдера:

```php
$events->listen(FooHappened::class, [HandleFooListener::class, 'handle']);
```

### 4.7. HTTP-контроллер

`__invoke` или один метод, инъекция через сигнатуру:

```php
final class SubmitFooController extends Controller
{
    public function __invoke(SubmitFooRequest $request, SubmitFooHandler $handler): JsonResponse
    {
        $handler->handle($request->toCommand());
        return response()->json(['ok' => true]);
    }
}
```

`FormRequest` владеет `rules()`, `messages()`, `attributes()`, `withValidator()` (доп. проверки типа капчи) и обязательным `toCommand(...): SubmitFooCommand`. См. `SubmitReviewRequest.php`.

### 4.8. Async job

```php
final class SendXxx implements ShouldQueue
{
    use Queueable;
    public int $tries = 10;
    public function backoff(): array { return [30, 60, 120, 300, 600, 1200, 1800, 3600]; }
    public function __construct(public string $id) {}
    public function handle(SomeHandler $h): void { /* достать модель -> вызвать use case */ }
}
```

Job — не место для бизнес-логики, только склейка. Дёргать use case.

## 5. Маршрутизация и middleware

- `routes/api.php` — публичные API под префиксом `/api/public/...`, всё за middleware:
  - `tenant` (`ResolveTenantBySubdomain`) — резолвит `Owner` по поддомену, кладёт в `$request->attributes->get('tenant')`.
  - `subscription.active` (`EnsureSubscriptionActive`) — отбрасывает запрос если подписка истекла.
  - `resolve.public.place` (`ResolvePublicPlace`) — кладёт доменный `Place` в `$request->attributes->get('resolved_place')`.
  - `throttle:10,1` на отзывы.
- Webhooks: `POST /api/webhooks/telegram`, `POST /api/webhooks/tinkoff` — без префикса/middleware (свои механизмы аутентификации).
- `routes/web.php` — корневой healthcheck, `/payment/{success,fail}`, `/privacy`.
- `routes/telegram.php` — биндинги Nutgram. Команды/conversations/callbacks ссылаются на классы из `App\Interface\TelegramBot\*`.
- Middleware-алиасы регистрируются в `bootstrap/app.php`.

**Ошибки API** возвращаются через `App\Support\ApiResponse::error(ApiErrorCode::X, $status)` — единый формат `{message, code}`. Новые коды — в `App\Enums\ApiErrorCode`.

## 6. Уведомления

- Контракт `OwnerNotifier` (`MultiChannelOwnerNotifier`) шлёт через **instant-каналы** (Telegram → MAX), при провале — **fallback** (Email).
- Добавить новый канал = реализовать `App\Application\Notifications\Channels\NotificationChannel` (`supports`, `deliver`) и вставить в массив в `NotificationsServiceProvider`.
- `AdminNotifier` — для критических ошибок (`AlertAboutCriticalError`).
- DTO уведомления: `App\Application\Notifications\OwnerNotification` + `NotificationAction[]`.

## 7. Платежи

- Use cases: `InitSubscriptionPayment` (создаёт `PaymentTransaction`, идёт в `AcquirerGateway` за платёжной ссылкой), `HandlePaymentNotification` (идемпотентен, продлевает подписку и шлёт `ConfirmSubscriptionRenewed`).
- Порты: `AcquirerGateway`, `PaymentNotificationParser` — реализованы для Tinkoff в `Infrastructure/Payments/Tinkoff/`.
- Подмена эквайера = новые имплементации портов + замена биндингов в `PaymentsServiceProvider`.
- **Webhook должен быть идемпотентным** — `transaction->isFinalized()` ранний return.

## 8. Миграции и Eloquent-модели

- Все агрегаты — UUID PK (`HasUuids`).
- Миграции — `database/migrations/`, формат `YYYY_MM_DD_HHMMSS_<desc>.php`, имена в snake_case.
- Eloquent-модель имеет атрибут `#[Fillable([...])]` и `protected function casts(): array` (включая `enum::class` для статусов).
- Маппер живёт рядом с репозиторием: `Infrastructure/Persistence/Eloquent/<Context>/<Aggregate>Mapper.php`.

## 9. Тесты

- **`tests/Unit/Domain/...`** — чистый PHP, без БД, без `RefreshDatabase`.
- **`tests/Unit/Application/...`** — in-memory fake-репозитории (анонимные классы), fake `Clock`, fake `IdGenerator`, fake `DomainEventDispatcher`. См. `tests/Unit/Application/Reviews/SubmitReviewHandlerTest.php` как эталон.
- **`tests/Feature/...`** — HTTP/Telegram/DB сценарии, фокус на бизнес-результате.
- Хелперы — `tests/Helpers/{iam,places,reviews,payments,telegram,queue}.php`.
- Запуск: `composer test`, покрытие — `composer test:coverage`.

При добавлении фичи — **минимум 1 unit-тест на handler и 1 на агрегат**, при HTTP-фиче — feature-тест на маршрут.

## 10. Checklist: «добавить новую фичу X»

1. **Определи контекст.** Если ни один не подходит — заведи новый каталог `Domain/<Context>/` + `Application/<Context>/` + `Infrastructure/Persistence/Eloquent/<Context>/` + `<Context>ServiceProvider.php` (зарегистрировать в `bootstrap/providers.php`).
2. **Domain.** Дополни/создай агрегат, VO, enum. Если есть последствия в других контекстах — Domain Event.
3. **Application.** Создай папку `<UseCase>/` с `Command`/`Query` + `Handler` (+ `Result`/`View` при необходимости).
4. **Infrastructure.** Eloquent-модель (если новая таблица) + миграция + маппер + репозиторий/ридер + биндинг в сервис-провайдере.
5. **Listeners** (если есть Domain Events) — в `Application/<Context>/Listeners/`, регистрация в `boot()` провайдера.
6. **Interface.** Контроллер/FormRequest/Telegram-команда — тонкий, переводит ввод в Command и вызывает Handler.
7. **Routes.** `routes/api.php` (HTTP) или `routes/telegram.php` (бот) или `routes/console.php` (cron). Middleware `tenant`/`subscription.active`/`resolve.public.place`/`throttle` по ситуации.
8. **Async** — если работа дорогая или внешняя: `App\Jobs\<Name>` с `tries`/`backoff`, диспатчится из листенера или handler'а.
9. **Тесты.** Unit на агрегат + Unit на handler. Feature если меняется внешний API.
10. **Проверь:**
    - в Domain нет `use Illuminate\...`;
    - VO/Command/Handler помечены `final readonly`;
    - есть `declare(strict_types=1);`;
    - русские сообщения об ошибках для пользовательских (валидация, домен) — UI на русском;
    - идемпотентность для webhook'ов и платёжных операций;
    - запусти `composer test` и `./vendor/bin/pint` (форматирование).

## 11. Подводные камни

- **Eloquent-модель из чужого контекста.** Если нужны данные из другого контекста — иди через его use case (или Reader), не через `ChildModel::query()->where(...)`.
- **`now()` / `Carbon::now()` в Domain/Application.** Запрещено. Только `Clock $clock` в конструкторе.
- **Тяжёлая логика в Job.** Job — это адаптер очереди. Бизнес-логика — в Application/Handler, который Job дёргает.
- **Throw в `NotificationChannel::deliver()`.** Каналы не должны кидать — MultiChannelOwnerNotifier полагается на тихий фейл + fallback.
- **Migration без UUID.** Все новые сущности с PK = `$table->uuid('id')->primary();` + FK `foreignUuid(...)`.
- **Тест без `RefreshDatabase`.** Application-тесты делают fake-репо в памяти; БД-туда не нужна. `RefreshDatabase` — только для Feature.
- **Логика в контроллере.** Если в `__invoke` больше 5-7 строк или есть `if (бизнес-правило)` — это запах. Перенеси в handler.
- **Tinkoff URL по http.** В Docker reverse-proxy (Caddy/Traefik) разбирается trustProxies, иначе signed URL ломаются. Уже настроено в `bootstrap/app.php`, не трогать.
- **Default-тариф ищется по `tariffs.is_default = true`** (миграция `2026_05_25_120000_…`). До этой миграции — по `title='MVP'`. `TariffSeeder` ставит `is_default=true` для MVP. `TariffFactory` намеренно НЕ ставит `is_default` — иначе любая фабрика создавала бы «второй default». Если нужен default в feature-тесте — либо `$this->seed(TariffSeeder::class)`, либо явный `is_default => true` в `Tariff::factory()->create([...])`. Инвариант «ровно один is_default» обеспечивает `SetDefaultTariffHandler` через `TariffRepository::markAsOnlyDefault()`.

## 13. Админ-панель (Filament v5)

URL `/admin`, отдельный guard `admin` (один супер-админ из `.env` через `EnvUserProvider`, без таблицы `admin_users`). Подробный handoff и подводные камни — в **`backend/админка-план.md`**.

Карта Resource'ов:

| Resource | Модель | Особенности |
|---|---|---|
| `Owners/OwnerResource` | `User` | Все мутации через use cases (UpdateOwnerProfile/DeleteOwner/OverrideSubscription/ChangeOwnerTariff/ExtendSubscription/IssueOwnerImpersonationToken). |
| `Places/PlaceResource` | `Place` | Repeater платформ (`PlaceForm::normalizeRepeaterPlatforms` → use case). QR preview/download через `App\Services\QrCodeService` + `PlaceMapper::toDomain($eloquent)`. Bulk activate/deactivate. |
| `Reviews/ReviewResource` | `Review` | Без `Create`-страницы. EditReview = только статус (AdminChangeReviewStatusHandler). Delete с textarea-reason → DeleteReviewHandler (пишет в `action_logs` тип AdminDeletedReview перед удалением). `ResendNegativeReviewAlert` через `SendNegativeReviewAlert::dispatch`. |
| `Tariffs/TariffResource` | `Tariff` | **Исключение из правила «всё через use cases»**: конфигурационная сущность без сложных инвариантов → CRUD напрямую через дефолтные `CreateRecord`/`EditRecord`. `set_default` — единственное действие через use case (`SetDefaultTariffHandler`, гарантирует «ровно один is_default» атомарно через `TransactionRunner`). |
| `PaymentTransactions/PaymentTransactionResource` | `PaymentTransaction` | **Read-only** (`canCreate()=false`, только `index`/`view`). Actions: `refire_webhook` (собирает подписанный Tinkoff-payload → `HandlePaymentNotificationHandler`, `visible(!app()->isProduction())`), `mark_failed` (`ForceFailPaymentHandler`, виден только для Pending). |

**Правило**: Filament Resource = Interface-слой, такой же тонкий, как HTTP-контроллер.
- Чтение (`table`/`infolist`) — напрямую через Eloquent.
- Запись (любые мутации) — через `app(<UseCase>Handler::class)->handle(...)`.
- Кастомный admin-action со внешним эффектом (webhook, тестовая отправка) — `->visible(fn () => ! app()->isProduction())`.

Тесты — `tests/Feature/Admin/*ResourceTest.php` (HTTP smoke + интеграция handlers поверх Eloquent) и `*ResourceFiltersTest.php` (Livewire-фильтры через `Filament::setCurrentPanel('admin')`).

## 14. Owner-панель (React SPA на `{slug}.otziv.space/owner`)

Отдельный фронт для конечных владельцев заведений. Полный план и handoff —
**`backend/docs/owner-panel-plan.md`** (с текущим статусом фаз). Не путать с
Filament-админкой: разные guard'ы, разные цели, разные стили.

**Стек:** React 18 + Vite + TypeScript + Tailwind + TanStack Query + Zustand + PWA.
Стиль — Kvell Merchant (светлая палитра, soft shadows, rounded-2xl, mobile-first).

**Архитектурные правила (строго!) — см. `backend/docs/owner-panel-plan.md` § 0.1:**
- **FSD (Feature-Sliced Design):** слои `app → pages → widgets → features → entities → shared`.
  Импорты только вниз. Кросс-слайс — только через public API (`<slice>/index.ts`).
  Enforce: `eslint-plugin-boundaries`.
- **Только функциональное программирование:** никаких `class` в нашем коде. State — hooks/Zustand;
  pure-функции — в `<slice>/lib/`.
- **Server state — только TanStack Query.** Прямой `fetch`/`axios.get` в `ui/`/`pages/`/`widgets/`
  запрещён. Любой сетевой вызов = `useXxxQuery`/`useXxxMutation` в `<slice>/api/`.
- HTTP-клиент один — `shared/api/httpClient`, скрыт внутри React Query hooks.

> ⚠️ Каркас Фазы 0 заскаффолжен функционально (тесты зелёные), но структурно
> НЕ соответствует FSD. План миграции (Фаза 0.5) — в `owner-panel-plan.md` § 1.6.
> Прежде чем браться за Фазу 1, выполнить миграцию.

**Расположение:** `frontend/owner/`, workspace в `frontend/package.json`.
Команды:
- `npm run dev:owner` — Vite dev-server на :5174 (с proxy `/api`, `/sanctum` → :8000).
- `npm run build:owner` — собирает в `dist/owner/index.html` (его читает бэк).
- `npm run test --workspace owner` — vitest.

**Backend под Owner-панель:**
- `OwnerSpaController` (`app/Interface/Http/Controllers/Owner/`) отдаёт SPA-shell
  под middleware `tenant`. Без тенанта в поддомене → 404.
- `routes/web.php` — catch-all `Route::middleware('tenant')->get('/owner/{any?}', ...)`.
- `routes/api.php` — группа `Route::prefix('owner')->middleware('tenant')` для
  будущих `/api/owner/*` endpoint'ов. В Фазе 0 — только `me` (заглушка 401).
- `config/auth.php` — новый guard `owner` (driver=session, provider=users).
- `config/sanctum.php` — `'guard' => ['owner', 'web']` (cookie-auth для SPA).
- `config/cors.php` — `supports_credentials=true`, allowed origins из env.
- `.env.example` — `SANCTUM_STATEFUL_DOMAINS` (включая `*.otziv.space`),
  `CORS_ALLOWED_ORIGINS`.

**Тенант резолвится через существующий `ResolveTenantBySubdomain`** — это та же
машина, которой пользуются публичные scan-API. В тестах поддомен передаётся
через `X-Tenant-Slug` header (см. `TenantResolver`).

**Аутентификация (планируется в Фазе 1):** magic-code через Telegram-бот.
Owner пишет `/login` боту → бот возвращает inline-кнопку с deep link
`{slug}.otziv.space/owner/login?code=…` → SPA меняет code на cookie-сессию
через `POST /api/owner/auth/exchange`.

**Тесты:** `tests/Feature/Owner/OwnerSpaTest.php` (5 кейсов: SPA-shell под
тенантом, catch-all fallback, без тенанта = 404, без бандла = 404, `/api/owner/me`
без сессии = 401).
